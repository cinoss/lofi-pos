# Vietnamese Localization (Lingui) — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Wire Lingui v5 into all three React apps (cashier, admin, web) and packages/ui. Default locale `vi`, fallback `en`. Catalogs in PO format with TypeScript-compiled output. Locale picked from the venue's `locale` setting at startup; when user switches it in admin, all clients pick it up after refresh. Add a workspace `justfile` with extract/compile commands.

**Architecture:**
- One Lingui catalog per app (cashier / admin / web) — apps import macros and reference their own catalog
- Shared strings in `packages/ui` and `packages/pos-ui` get extracted into the consuming app's catalog (Lingui treats source files holistically; we point each app's `include` glob at its own src + the workspace packages it consumes)
- `i18n` singleton per app at `apps/<app>/src/lib/i18n.ts` — loads compiled catalogs, activates the requested locale, exposes `setLocale`
- App startup: fetch `Settings.locale` from server → activate matching catalog (default `vi`)
- Vite's `@vitejs/plugin-react` configured to apply `babel-plugin-macros` so Lingui's `t\`...\`` and `<Trans>` macros transform at build time
- `justfile` at repo root collects helper commands: `just i18n-extract`, `just i18n-compile`, scoped per-app variants

**Out of scope (separate plans):**
- Translating every string in the codebase. This plan wires infrastructure and translates the most-visible screens (setup wizard, login, lock, app shell nav, sessions list, payment); others get translated incrementally.
- Per-user locale (it's a venue-wide setting; staff don't pick).
- RTL or non-Latin scripts beyond Vietnamese.
- Currency formatting (separate concern; would use `Intl.NumberFormat` with the venue's `currency`).
- Date/time formatting beyond Lingui's built-in `<Plural>` / `i18n.date(...)`.


Decisions:
- **Lingui v5** (matches reference). PO format via `@lingui/format-po`. TypeScript-compiled catalogs.
- **`babel-plugin-macros`** invoked through `@vitejs/plugin-react`'s `babel` option — minimal config, no need for `@lingui/swc-plugin` (which is also valid but requires SWC).
- **Default locale is `vi`** (project audience). Source locale is `en` (developer language; macros expect English source strings).
- **Locale lives in venue settings** (`locale: 'vi-VN'` already added in the wizard plan). Lingui only cares about the language tag — strip the region: `vi-VN` → `vi`.
- **Per-app catalogs** — keeping separate catalogs avoids cross-app collisions and makes it obvious where a string lives. Translators get three small catalogs rather than one giant.
- **Workspace package strings** are extracted into the consuming app's catalog. The same `<Trans>Sessions</Trans>` in `packages/pos-ui/src/components/app-shell.tsx` ends up in cashier, admin, and web catalogs. Translators do it once per app — small duplication, clear ownership.
- **`justfile` at repo root** with one line per common task. New file; we don't have one yet. Helpers cover extract + compile per app and a workspace-wide combo.
- **Translation discovery** uses Lingui's `lingui extract` which scans the `include` glob and updates PO files in place. CI can later add `lingui extract --clean --strict` to fail on unknown strings.

```ts
// apps/cashier/src/lib/i18n.ts
import { i18n } from "@lingui/core";
import { messages as en } from "../locales/en/messages";
import { messages as vi } from "../locales/vi/messages";

export const DEFAULT_LOCALE: SupportedLocale = "vi";
export type SupportedLocale = "en" | "vi";

i18n.load({
  en,
  // vi inherits from en for any not-yet-translated strings
  vi: { ...en, ...vi },
});
i18n.activate(DEFAULT_LOCALE);

/** Activate a new locale; safe to call repeatedly. */
export function setLocale(locale: SupportedLocale) {
  i18n.activate(locale);
}

/** Convert a BCP-47 tag (e.g. "vi-VN") to a supported locale. */
export function normalizeLocale(tag: string | null | undefined): SupportedLocale {
  if (!tag) return DEFAULT_LOCALE;
  const lang = tag.split(/[-_]/)[0]?.toLowerCase();
  return lang === "en" ? "en" : "vi";
}

export { i18n };
```
